## Paragraph 1 — Concrete Learning Moment: The Accuracy That Lied

The clearest thing I learned in this course came from a chain of failures in Week 6. My ANN on the Truth Seeker Dataset reported 99.73% test accuracy after the first run, and my initial reaction was satisfaction — the model was working. Looking more carefully at the code, I noticed that both TF-IDF vectorizers were being fit on the entire 134,198-row dataset before the train/test split, which meant the vocabulary statistics were computed using tokens that would later appear in the "unseen" test set. I corrected the pipeline so that all fitting — TF-IDF and StandardScaler — happened on training data only, with the test set receiving a transform-only pass. Run 2 produced 99.72% accuracy: a change of 0.01 percentage points. The fix had done almost nothing. That forced me to look harder at the dataset structure itself, where I found that the 134,198 rows map to only 1,058 unique fact-checked statements, with each statement repeated dozens of times across different tweets. A row-level 80/20 split does not separate statements — it splits the same statement's rows between train and test, so the model memorizes statement vocabulary and appears to generalize when it is actually pattern-matching text it has already seen. Switching to a statement-level split — assigning all rows for a given statement entirely to train or entirely to test — dropped Model 1 accuracy to 77.22%. That number is the honest one. What I now understand, in a way I did not before this lab, is that accuracy is only a meaningful metric if the test set is genuinely unseen in the way that matters for the task. Here, "unseen" had to mean unseen *statements*, not unseen rows.

## Paragraph 2 — Meaningful Difficulty: Deciding What a Number Means

The hardest part of Week 6 was not technical — it was a judgment call about Model 2. After the statement-level split, Model 2 (User Behavior features) showed 50.84% accuracy. That number sits just above random chance, and one response would have been to tune the model further: adjust learning rate, add features, run more epochs. But looking at the confusion matrix made the interpretation unavoidable. The model predicted "True" for 25,636 out of 26,268 test samples — 97.2% of actual false claims were misclassified as true, and the recall for the false class was 2.8%. This was not a model that was learning slowly; it was a model that had collapsed to always predicting the majority class. The difficulty was choosing between two honest conclusions: either the model needed more development, or the nine user behavior features — follower counts, bot scores, engagement ratios — simply carry no meaningful signal for distinguishing true from false claims on statements the model has never seen. The training curves confirmed the second interpretation. Validation accuracy was flat between 2–5% from epoch 1, and validation loss never decreased across all 50 epochs. I reported Model 2 as a failure rather than a work in progress, because the evidence was conclusive: the features lacked predictive power, not the architecture. Learning to distinguish "this model needs more work" from "this hypothesis is wrong" is the kind of judgment that does not come from running code — it comes from reading what the outputs are actually saying.

## Paragraph 3 — AI Use: Useful Until It Isn't

Across the labs in this course I relied on AI heavily for code scaffolding — generating boilerplate sklearn pipelines, matplotlib visualizations, and debug loops that would have taken significantly longer to write from scratch. The Week 2 AI usage statement documents this honestly: code generation, visualization formatting, debugging encoding errors. But the most useful thing I learned about AI-assisted coding came from noticing what the AI got wrong. In the Week 2 PCA code, the AI-generated pipeline called `scaler.fit_transform(X_raw)` on the full dataset before any split had taken place. The code ran without errors, produced clean output, and looked correct — and it was wrong. The same pattern appeared in the first version of the Week 6 ANN pipeline, where TF-IDF vectorizers were fit on the full dataset before the split. Both cases involved AI generating statistically standard-looking preprocessing code that violated the train/test separation principle. The error is invisible to a syntax checker and passes silently at runtime; it only becomes visible when you ask whether the logic matches the underlying concept. After encountering this twice, I added an explicit checkpoint to any preprocessing pipeline: identify every `fit` or `fit_transform` call and confirm it is executed on training data only. That habit — verifying AI output against the conceptual requirement, not just the execution result — is the most transferable thing I am leaving this course with.
